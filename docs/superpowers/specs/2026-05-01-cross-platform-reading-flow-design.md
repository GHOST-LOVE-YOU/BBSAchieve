# 双端只读阅读链路设计

## 概述

这份设计聚焦前端下一阶段的最小功能闭环：同时完善 Web 与移动端的公开阅读能力，在不做正式 UI 设计的前提下，先打通“首页版面列表 -> 版面帖子列表 -> 帖子详情与回复列表”的只读链路。

这份设计建立在现有前端数据底座设计之上，继续使用前端本地假数据与内存仓储，不直接接入真实后端同步 API。目标是先把双端共用的读取边界、页面职责和测试契约做稳定，为后续替换为真实数据源保留清晰接口。

这份设计不覆盖：

- 正式视觉设计与样式系统
- 发帖、回帖、编辑、删除等写操作
- 通知订阅能力
- 搜索、筛选、分页等扩展浏览能力
- 真实后端同步接入

## 设计结论

这一轮采用“共享读取用例 + 各端各自消费页面”的方案。

- `packages/domain` 继续承载论坛实体和仓储接口，不引入页面层查询逻辑。
- `packages/state` 新增面向阅读链路的只读查询用例，负责把仓储数据整理成页面可直接消费的结果对象。
- `packages/test-utils` 提供这轮功能所需的假数据构造器与内存仓储实例，确保 Web 与移动端读取的是同一套沙盒论坛数据。
- `apps/web` 完善公开区页面：首页列出版面，版面页列出帖子，帖子页展示正文与回复列表。
- `apps/mobile` 建立同一条阅读链路的最小页面，只保留基础 `View`/`Text` 结构，不提前引入复杂视觉或交互包装。

这个方向的核心价值是把“读取什么”和“如何展示”分开。当前虽然使用内存仓储，未来切换为真实 API 或本地数据库时，主要替换共享读取依赖，而不是让 Web 和移动端页面分别重写数据获取逻辑。

## 架构与职责边界

### `packages/domain`

保持当前职责不变：

- 提供 `Board`、`Thread`、`Reply`、`User` 等领域实体与 schema
- 定义读取所依赖的仓储接口
- 不承担页面聚合结果和展示语义

### `packages/state`

新增阅读链路查询用例，负责：

- 调用仓储接口读取版面、帖子、回复数据
- 聚合页面需要的派生字段，例如帖子数量、回复数量、最新帖子标题
- 统一返回明确的查询结果语义，例如 `success`、`notFound`、`error`

页面层不应直接操作底层仓储或拼装数据。

### `packages/test-utils`

承担这轮功能的共享测试数据基础设施：

- 提供论坛假数据构造器
- 提供用于阅读查询测试的内存仓储实例
- 保证双端测试和共享用例测试依赖同一套数据语义

### `apps/web`

公开区页面职责：

- 首页读取版面摘要列表并提供进入版面的入口
- 版面页读取单个版面详情及帖子列表
- 帖子详情页读取帖子正文、所属版面信息和回复列表

Web 页面只消费查询结果，不直接内联假数据。

### `apps/mobile`

移动端页面职责与 Web 保持一致：

- 首页列出版面入口
- 版面页列出帖子摘要
- 帖子页展示正文与回复

移动端这一步只保证阅读链路成立，不在本轮加入复杂导航花样或视觉组件体系。

## 读取用例设计

### `getBoardSummaries()`

用于首页版面列表。

返回结果应至少包含：

- `id`
- `slug`
- `name`
- `description`
- `threadCount`
- `latestThreadTitle`

首页仅依赖这些摘要字段，不直接读取完整帖子内容。

### `getBoardDetail(boardIdOrSlug)`

用于版面页。

返回结果应至少包含：

- 版面基本信息
- 帖子摘要数组

每条帖子摘要至少包含：

- `id`
- `title`
- `authorName`
- `replyCount`
- `publishedAt`

如果版面不存在，返回明确的 `notFound` 结果，而不是用空数组表达异常。

### `getThreadDetail(threadId)`

用于帖子详情页。

返回结果应至少包含：

- 帖子标题
- 帖子正文
- 作者名称
- 发布时间
- 所属版面摘要
- 按展示顺序排好的回复数组

回复项当前只承担只读展示，不带新增、编辑、删除等动作语义。

## 页面链路

### 首页

首页展示版面列表，用户从首页进入某个版面。

首页不承担：

- 搜索
- 排序切换
- 推荐区块
- 最近回复流

### 版面页

版面页展示：

- 版面名称
- 版面描述
- 该版面的帖子列表

用户从帖子列表进入帖子详情页。

### 帖子详情页

帖子详情页展示：

- 帖子标题
- 帖子正文
- 作者信息的最小文本表达
- 回复列表

回复只读展示，不提供回复输入框或任何写操作入口。

## 异常与空态边界

为了让未来切换真实数据源时页面契约保持稳定，这一轮就明确约束空态和异常语义。

### 首页版面列表

- 有数据时展示版面列表
- 无数据时展示“暂无版面”

### 版面页

- 版面存在但无帖子时展示“暂无帖子”
- 版面不存在时走 `notFound` 结果

### 帖子详情页

- 帖子存在但无回复时展示“暂无回复”
- 帖子不存在时走 `notFound` 结果

### 错误处理

内存仓储阶段不引入复杂重试逻辑，但查询用例仍保留 `error` 结果语义，为未来真实数据源接入预留稳定边界。

## 非目标与范围收束

这一轮明确不做以下能力，避免阅读链路被额外交互打散：

- 新增回复
- 发帖
- 编辑内容
- 作者主页
- 搜索与筛选
- 分页
- 回复折叠
- 来源恢复或复杂返回状态管理

这保证当前阶段聚焦在双端共用的只读阅读链路。

## 测试策略

### `packages/state`

这是本轮最核心的测试层。

需要覆盖：

- `getBoardSummaries()` 正常返回版面摘要列表
- `getBoardDetail()` 正常返回版面与帖子列表
- `getThreadDetail()` 正常返回帖子正文与回复列表
- 版面不存在
- 帖子不存在
- 空版面
- 无回复帖子

### `apps/web`

做轻量公开路由测试，确认：

- 首页能渲染版面列表
- 版面页能渲染帖子列表
- 帖子页能渲染正文与回复
- `notFound` 场景有明确输出

不测试视觉样式细节。

### `apps/mobile`

做最小页面测试，确认：

- 首页展示版面入口
- 版面页展示帖子摘要
- 帖子页展示正文与回复

同样只验证阅读链路，不验证视觉层表现。

## 实施结果标准

完成本轮后，应能满足以下结果：

1. Web 用户可以从首页进入版面，再进入帖子详情并看到回复列表。
2. Mobile 用户可以通过同一套共享读取逻辑完成相同阅读链路。
3. 双端页面不直接依赖散落的假数据文件，而是统一通过共享查询用例读取。
4. 阅读链路的空态、`notFound` 和基础测试边界已经被明确定义。

这将作为下一步编写实现计划的依据。
