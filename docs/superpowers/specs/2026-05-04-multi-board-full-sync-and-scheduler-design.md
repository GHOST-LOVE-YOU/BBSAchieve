# 多板块全量抓取与独立定时同步设计

## 概述

这份设计聚焦当前下一阶段的采集能力建设：

1. 除 `IWhisper` 外，为其它目标板块补上一轮当前可见内容的全量抓取与存储。
2. 为这些板块提供类似“从旧库导入 iwhisper”的 Admin 手动任务入口。
3. 为这些板块提供各自独立、时间不一致的定时同步任务。
4. 在整个抓取链路中加入更明确的限速与串行约束，避免影响北邮人主站。

这轮设计默认旧数据库中的 `iwhisper` 历史内容已经全部导入新库，因此“从旧库导入 iwhisper”只作为历史实现参考，不再继续保留为产品入口。

## 已知前提

当前仓库已经具备以下基础：

- `backend` 已具备北邮人登录、板块页抓取、帖子页抓取和 `/api/sync/updates`、`/api/sync/backfill` 同步接口。
- `apps/web` 已具备通过 `runByrSyncImport` 调用同步接口、整理 payload 并写入本地 PostgreSQL 的导入链路。
- `apps/web` 已具备进程内定时任务框架、定时任务运行历史表和 Admin 定时任务页面。
- `apps/web` 已具备长任务型的 Admin 入口与 `import_jobs` 表，当前主要用于旧库 `iwhisper` 迁移。

本轮设计在此基础上推进，不单独创造第二套抓取架构。

## 覆盖目标

本轮设计覆盖：

1. 多板块硬编码配置方式。
2. 非 `IWhisper` 板块的一次性全量抓取入口。
3. 每板块独立定时任务的定义方式。
4. 全局串行限速与冲突处理语义。
5. Admin 页面如何展示与触发这些任务。
6. 旧 `legacy iwhisper` 入口退场后的替代结构。

本轮设计不覆盖：

- 从北邮人首页自动发现全部板块。
- 在线新增、删除或修改板块配置。
- 队列化排队执行。
- 多实例分布式锁或主实例选举。
- 精确的请求中断能力。
- 新增独立“全量抓取接口”。

## 设计结论

本轮设计采用以下方向：

- 目标板块列表硬编码在代码中，不从首页自动发现。
- 每个目标板块的首次全量抓取由 Admin 手动触发，一板块一任务。
- 每个目标板块的后续同步继续走定时任务，但频率、窗口、启用状态都单独硬编码。
- 首次全量抓取不新增后端接口，直接复用现有 `/api/sync/updates` 这条同步链路。
- 首次全量抓取采用“单次超大时间窗口请求”的方式完成，而不是引入页级分段协议。
- 所有抓取流程共用一个全局串行节流器，同一时刻只允许一个抓取流程访问主站。
- 发生冲突时不排队：
  - 定时任务记 `skipped`
  - 手动全量任务记 `paused` 并附带跳过原因
- `legacy iwhisper` 的 Admin 入口移除，`import_jobs` 转型为通用长时导入任务表。

## 为什么选择单次超大窗口全量

当前有三种主要做法：

1. 在现有接口上补分段全量协议。
2. 对现有 `/api/sync/updates` 直接传一个足够大的 `window_minutes`，完成一次超长全量请求。
3. 新增独立全量接口。

本轮选择第 `2` 种。

原因如下：

- 你已经明确要求尽量复用后端代码，不单独创建新接口。
- 当前最需要的是尽快把“其它板块也能抓进来”这条链路跑通，而不是先设计复杂恢复协议。
- 现有 `runByrSyncImport` 和 `/api/sync/updates` 已经能支撑“Web 调后端、后端抓北邮人、Web 导入本地数据库”这条主链路。
- Admin 对首次全量的核心诉求是“有入口、有状态、有报错”，不是页级精细进度。

同时，这个选择也带来明确代价：

- 单个板块全量抓取会变成一次很长的请求。
- 任务进度只能做到粗粒度，不承诺页级游标恢复。
- 任务失败后，下一次重试通常是整板块重跑。
- 真正的抓取节奏控制必须更多下沉到 `backend` 请求循环内部。

这些代价属于本轮设计的已知取舍，不是假装不存在。

## 统一板块配置

### 配置原则

目标板块列表是固定的，因此本轮不做首页发现，不做数据库配置中心，而是统一硬编码在代码里。

建议新增一份面向 `apps/web` 的板块同步注册表，例如：

- `apps/web/src/server/boardSync/boardRegistry.ts`

每个板块至少定义以下字段：

- `boardName`
- `boardSlug`
- `title`
- `fullSyncEnabled`
- `fullSyncWindowMinutes`
- `scheduledSyncEnabled`
- `scheduledIntervalMinutes`
- `scheduledWindowMinutes`
- `description`

其中：

- `fullSyncWindowMinutes` 用于首次全量抓取时传给 `/api/sync/updates`
- `scheduledWindowMinutes` 用于后续定时增量同步
- 两者都属于代码事实源，不从数据库读取

### 与现有定时任务注册表的关系

当前 `apps/web/src/server/scheduler/taskRegistry.ts` 只注册了 `IWhisper`。

本轮建议保留“定时任务注册表”这层，但让它改为由统一板块注册表派生：

- `IWhisper` 继续保留原有任务定义
- 其它板块根据板块注册表生成各自的定时任务定义

这样可以避免：

- 板块定义写一份
- 定时任务定义又重复写一份

## 职责边界

### Backend 负责什么

`backend` 继续负责：

- 登录态管理与 cookie 复用
- 板块页抓取与帖子页抓取
- `/api/sync/updates` 与 `/api/sync/backfill`
- 在窗口同步语义下决定翻页与停止
- 在抓取循环内部执行请求节流

本轮不要求 `backend` 新增独立“全量抓取接口”。

对于非 `IWhisper` 板块的首次全量，`backend` 仍然只看到：

- `board_name`
- 一个非常大的 `window_minutes`

它继续沿用现有同步逻辑处理，只是在实现层需要保证：

- 超大窗口时仍然会继续正常翻页
- 在每次板块页或帖子页请求之间执行固定等待
- 不因为单次请求过长而失去最基本的错误反馈

### Web 负责什么

`apps/web` 负责：

- 硬编码目标板块清单
- 决定哪个板块允许首次全量
- 决定哪个板块启用定时同步
- 决定每个板块的同步频率与时间窗口
- 提供 Admin 手动入口
- 记录手动全量任务状态
- 记录定时任务执行历史
- 执行全局串行节流

`apps/web` 不负责：

- 解析北邮人页面 DOM
- 实现第二套后端全量接口
- 做跨实例分布式调度

## Admin 与数据模型

### `legacy iwhisper` 入口退场

“从旧库导入 iwhisper”已经完成历史目标，因此本轮应从 Admin 页面移除该入口。

这意味着：

- `/admin/imports` 不再显示“从旧库导入 iwhisper”按钮
- 旧 `legacy iwhisper` 任务页表格改为展示“板块全量抓取任务”
- 已存在的历史任务记录可以保留在数据库里，但不再作为当前产品能力继续暴露

### 复用 `import_jobs`

本轮不新增第二张长任务表，继续复用 `import_jobs`。

但其语义从“旧库迁移任务”扩展为“通用长时导入任务”。

建议新增并使用如下任务类型：

- `jobType = byr_board_full_sync`

同时约定：

- `sourceType = byr_sync_api`
- `sourceLabel = <boardName>`

`metadataJson` 至少记录：

- `boardName`
- `fullSyncWindowMinutes`
- `triggerSource`
- `requestedBy`
- `throttlePolicy`

### `import_jobs` 状态语义

第一版建议约定如下：

- `pending`
  - 任务已创建，尚未真正开始调用后端
- `running`
  - 已进入抓取流程，正在执行全量导入
- `paused`
  - 本次没有拿到全局抓取权，或人工要求后续再继续
- `failed`
  - 这次调用同步接口失败
- `succeeded`
  - 这次全量调用成功结束
- `cancelled`
  - 运营侧明确要求停止，不再把这次任务视为继续进行中

由于 `import_jobs` 没有 `skipped` 状态，本轮明确用：

- `paused + progressNote="skipped by global throttle"`

来表达“这次本想跑，但因为全局串行限速没有执行”。

本轮不要把这种“未获得抓取权”的情况误记成 `cancelled`。
`cancelled` 只保留给人工主动停止语义。

### `scheduled_task_runs` 继续保留独立职责

定时任务执行历史继续只记在 `scheduled_task_runs`，不混入 `import_jobs`。

原因：

- 手动全量任务和定时任务是两类不同心智模型
- 手动全量更接近“运维长任务”
- 定时任务更接近“周期执行记录”

当前表结构已经支持：

- `running`
- `succeeded`
- `failed`
- `skipped`

因此定时任务冲突时直接记录 `skipped` 即可。

### Admin 页面结构

`/admin/imports` 页面建议改成三块：

1. 手动导入入口区
   - 展示硬编码板块列表
   - 每个板块有“开始全量”“停止全量”按钮
2. 最近导入活动
   - 保留对 `imports` 和 `import_jobs` 的聚合展示
3. 板块全量任务表格
   - 展示每个板块最近任务状态、开始时间、结束时间、错误摘要

`/admin/scheduled-tasks` 页面继续保留，但其内容改成：

- `IWhisper` 既有任务
- 其它硬编码板块的独立定时任务

页面继续只展示代码定义与最近执行结果，不支持在线改配置。

## 执行流程

### 手动全量流程

单个板块的首次全量流程如下：

1. Admin 点击该板块“开始全量”。
2. `apps/web` 创建一条 `import_jobs` 记录。
3. runner 尝试进入全局抓取节流器。
4. 如果未取得抓取权：
   - 任务标记为 `paused`
   - `progressNote = "skipped by global throttle"`
   - 本次结束，不排队
5. 如果取得抓取权：
   - 任务标记为 `running`
   - 调用 `runByrSyncImport`
   - 传入目标板块名和该板块硬编码的超大 `fullSyncWindowMinutes`
6. `backend` 在同一条 `/api/sync/updates` 逻辑中执行抓取、翻页和限速。
7. Web 收到 payload 后复用现有 `mapSyncPayload`、`importSyncBatch` 写库。
8. 成功则标记 `succeeded`，失败则标记 `failed`。

### 定时增量流程

每个板块的定时同步继续沿用现有调度器模式：

1. 由代码注册表定义 `taskKey`
2. 由 `instrumentation.ts` 启动调度器
3. 到达间隔后触发该板块 runner
4. 尝试进入全局抓取节流器
5. 如果当前已有任何抓取流程在跑：
   - 直接创建一条 `scheduled_task_runs`
   - 状态记为 `skipped`
   - `skippedReason = "global throttle active"`
6. 如果取得抓取权：
   - 调用 `runByrSyncImport`
   - 使用该板块自己的 `scheduledWindowMinutes`
   - 成功或失败后写回 `scheduled_task_runs`

### 手动全量与定时任务的关系

两者在运营语义上相互独立。

这意味着：

- 某板块即使还没做过首次全量，也可以先跑定时同步
- 首次全量成功与否，不决定该板块定时任务是否允许存在
- 真正限制二者的只有：
  - 代码启用状态
  - 全局串行节流器

## 限速与串行策略

### Web 侧全局串行节流

第一层保护放在 `apps/web`。

目标是：

- 手动全量和定时同步共用一把“抓取锁”
- 同一时刻最多只允许一个抓取流程真的去调用 `backend`
- 冲突时不排队，直接落状态

建议新增一个独立的 server-only 抓取协调器，例如：

- `apps/web/src/server/syncThrottle/globalSyncThrottle.ts`

它至少提供：

- `tryAcquire()`
- `release()`
- 当前持有者信息

第一版只要求单实例内生效，不做跨实例共享。

### Backend 侧请求节流

第二层保护放在 `backend`。

原因是：

- 方案 2 下，首次全量是一次超长请求
- 真正的板块页翻页和帖子页补拉都发生在 `backend`
- 如果只在 `web` 侧节流，仍然无法控制单次请求内部的抓取频率

因此，`backend` 在 `/api/sync/updates` 的抓取循环中应加入固定等待策略，例如：

- 每次板块页请求之间等待固定毫秒数
- 每次线程快照补拉之间等待固定毫秒数
- 对异常重试次数保持克制

第一版建议把这些参数写成代码安全默认值，必要时允许环境变量覆盖，但不做在线配置。

## 停止、失败与重试语义

### 停止语义

因为首次全量是“单次超大窗口请求”，第一版不能承诺精确中断。

因此“停止全量”的定义是：

- Admin 点击停止后，把该任务标记为 `cancelled`
- 后续不再把它视为可继续中的正常任务
- 如果底层后端请求还在飞行，允许它自然结束
- 该飞行中的返回结果不应再把这条已取消任务覆盖为 `succeeded`

如果任务当时还没有真正取得抓取权，而只是停留在 `pending` 或 `paused`，也仍然直接记为 `cancelled`，表示运营侧明确放弃这次任务。

也就是说，第一版的停止更接近“运营停用”而不是“精确杀掉执行线程”。

### 失败语义

手动全量失败时：

- `import_jobs.status = failed`
- `errorMessage` 记录真实失败原因
- 下一次重试通常整板块重跑

定时任务失败时：

- `scheduled_task_runs.status = failed`
- 仅影响该次运行记录
- 不自动补排队

### 重试语义

本轮不设计页级恢复。

因此：

- 手动全量重试 = 再次发起一次新的整板块全量
- 定时任务重试 = 等下一个周期或手动点击“立即执行一次”

## 测试重点

第一版至少覆盖以下测试：

### Web 侧

1. 板块注册表可以正确生成定时任务定义。
2. `/admin/imports` 不再展示“从旧库导入 iwhisper”入口。
3. `/admin/imports` 可以展示硬编码板块的全量抓取入口。
4. 手动全量任务在节流器被占用时会记为 `paused` 且带跳过说明。
5. 定时任务在节流器被占用时会记为 `skipped`。
6. 停止后的任务不会被晚到的成功结果覆盖。

### Backend 侧

1. 超大 `window_minutes` 下仍然复用现有 `/api/sync/updates` 链路。
2. 抓取循环中的固定等待逻辑会生效。
3. 限速开启时不会改变已有 payload 结构。
4. 请求失败时能返回明确错误，而不是空成功。

## 分阶段落地顺序

虽然这份设计覆盖 `1 + 2 + 3`，但实现上仍建议分阶段推进：

1. 清理旧 `legacy iwhisper` Admin 入口，建立统一板块注册表。
2. 把单板块手动全量任务接入 `import_jobs`。
3. 在 `backend` 的同步抓取循环内补上请求节流。
4. 让板块注册表派生新的定时任务定义。
5. 补测试并验证手动全量、定时同步和冲突跳过语义。

这样可以先跑通“板块进入数据库”的主链路，再逐步把自动调度补齐。

## 需要明确保留的现实约束

最后有几条约束需要在实现前继续被视为事实：

- 北邮人历史内容可能继续消失，因此所谓“首次全量”只是“抓取当前仍可访问的全部内容”，不是历史永久补齐承诺。
- 其它板块列表在第一版是硬编码事实源，不依赖首页发现。
- 首次全量与定时同步都必须优先服从“不影响主站”的限速要求。
- 旧库 `iwhisper` 导入功能可以删除，不再继续维护同名产品入口。
