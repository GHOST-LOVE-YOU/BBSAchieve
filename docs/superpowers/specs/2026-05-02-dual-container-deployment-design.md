# Web 与同步后端双容器部署设计

## 概述

这份设计聚焦第三个子项目：为 Web 与 Python 同步后端提供可部署到云服务器上的独立 Dockerfile。

本轮设计覆盖以下目标：

1. 为 Web 提供 `Next.js standalone` 生产镜像方案。
2. 为 Python 同步后端提供独立生产镜像方案。
3. 明确 PostgreSQL 与 Redis 使用外部服务接入，而不是放进当前容器编排。
4. 明确运行时通过环境变量注入连接配置。
5. 保持仓库中的 spec、plan 与说明文档不记录真实连接串。

本轮设计不覆盖：

- `docker compose`
- Kubernetes
- Nginx、TLS 或域名接入细节
- 云平台运维细节
- PostgreSQL 与 Redis 的创建和托管方式

## 设计结论

部署子项目采用以下方向：

- 仓库提供两个独立 Dockerfile。
- `Web` 使用 `Next.js standalone` 生产镜像。
- `Python` 同步后端镜像直接运行现有 `byr-sync-api` 入口。
- PostgreSQL 与 Redis 都视为外部服务。
- 所有连接信息都通过运行时环境变量注入。
- 仓库中的 spec、plan、README 与示例命令不记录真实连接串。
- 真正执行部署前，通过对话向你确认运行时配置值。

## 为什么采用双容器 + 外部服务

你已经明确：

- Web 一个容器
- Python 后端一个容器
- PostgreSQL 不由当前任务负责
- Redis 也已经是外部实例
- 不需要 `docker compose`

因此当前最合理的部署边界是：

- 应用镜像负责打包和运行应用本身
- 数据与缓存服务通过环境变量接入外部实例
- 启动方式由你自行使用 `docker run` 控制

这样做的好处是：

- 与当前服务器实际资源形态一致
- 避免把真实连接配置硬编码进仓库
- 后续替换数据库或 Redis 地址时，不需要改镜像

## Web 容器设计

### 运行形态

Web 采用 `Next.js standalone` 生产镜像。

理由：

- 产物更集中，适合生产镜像构建。
- 镜像体积和运行依赖更可控。
- 与 Admin、公开阅读页和服务端接口共处一个运行时更自然。

### 镜像职责

Web 容器负责：

- 公开阅读页渲染
- `/admin` 管理界面
- Web 服务端接口
- Prisma 写库逻辑
- 旧库迁移后台任务
- 同步 API 导入编排

Web 容器不负责：

- 运行 PostgreSQL
- 运行 Redis
- 运行 Python 同步服务

### 运行时配置

Web 容器的数据库连接、同步 API 地址与鉴权信息都通过环境变量提供。

重要原则：

- 只在真正运行前向你询问具体值
- 不在 spec、plan 或提交到仓库的文档里记录真实地址
- 不在 Dockerfile 中写默认真实值

## Python 同步后端容器设计

### 运行形态

Python 容器直接运行现有 `byr-sync-api` 入口。

理由：

- 与当前代码结构最一致。
- 避免额外包一层自定义启动逻辑。
- 能直接复用当前后端 README 和入口语义。

### 镜像职责

Python 容器负责：

- 北邮人认证登录态管理
- 板块抓取与帖子抓取
- 同步 API 暴露
- Redis 短期同步缓存访问

Python 容器不负责：

- 写入新 PostgreSQL 展示库
- 执行 Web 管理台逻辑
- 执行旧库迁移任务

### 运行时配置

Python 容器运行时通过环境变量获取：

- 北邮人账号信息
- 同步 API token
- 外部 Redis 地址
- 其他必要后端配置

原则同样是：

- 真实值不写入仓库
- 真正执行前再向你确认

## 安全与配置边界

本轮设计明确要求：

- 真实数据库地址、旧库地址、Redis 地址、token、账号密码都不进入 spec
- 这些值不写进 plan
- 这些值不写进 README 示例
- 这些值只在真正运行前，通过对话向你确认

仓库里允许出现的只有：

- 环境变量名称
- 配置项说明
- 占位示例

不允许出现：

- 可直接连接生产或测试服务的真实连接串

## Dockerfile 设计原则

两个 Dockerfile 都应满足以下原则：

- 生产镜像尽量精简
- 构建阶段与运行阶段职责清晰
- 不把开发缓存、测试产物和无关文件带进最终镜像
- 不把本地 `.env` 和敏感配置打进镜像

### Web Dockerfile 原则

- 使用多阶段构建
- 先安装依赖并构建 Next.js
- 产出 standalone 运行目录
- 运行阶段只保留必要的 Node 运行环境与构建产物

### Python Dockerfile 原则

- 使用稳定 Python 生产基础镜像
- 安装运行 `byr-sync-api` 所需依赖
- 运行阶段只保留必要源代码与依赖
- 不在镜像内写死认证或缓存配置

## 运行说明边界

由于你不需要 `docker compose`，本轮产物只需要提供：

- 两个 Dockerfile
- 必要的运行说明
- 环境变量说明

具体 `docker run` 命令可以在真正部署时再按实际环境变量对话确认。

这能避免：

- 把真实连接配置写进仓库
- 提前固化可能很快变化的部署命令

## 与应用职责划分的关系

双容器部署与前面两个子项目的职责划分完全一致：

- Web 负责 Prisma 写库、Admin 界面与迁移任务
- Python 负责同步 API 与抓取缓存

因此部署设计不是额外重新定义架构，而是把已定下来的职责边界落实成可交付镜像。

## 验证重点

这一子项目完成后，至少应验证：

1. Web 镜像可以成功构建为 `Next.js standalone` 生产镜像。
2. Python 镜像可以成功构建并运行 `byr-sync-api`。
3. 两个镜像都不依赖本地 `.env` 硬编码。
4. Web 容器可以通过运行时环境变量连接新 PostgreSQL 并正常启动。
5. Python 容器可以通过运行时环境变量连接外部 Redis 并正常启动同步 API。
6. 仓库中的文档、Dockerfile 和计划文件没有包含真实连接串。

## 非目标

这份设计明确不定义：

- `docker compose`
- 反向代理和 HTTPS
- 容器编排平台
- 自动扩容
- 数据库与 Redis 的基础设施创建

这些都可以在应用镜像稳定后再独立规划。
