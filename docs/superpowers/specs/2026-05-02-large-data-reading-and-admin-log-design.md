# 大数据量下的帖子列表与 Admin 导入记录页设计

## 概述

这份设计聚焦第四个子项目：在真实数据大量导入后，优先保证帖子列表页和 Admin 导入记录页仍然可用、可扩展。

本轮设计覆盖以下目标：

1. 为公开帖子列表页定义第一版大数据量读取策略。
2. 为 Admin 导入记录页定义第一版读取与刷新策略。
3. 让数据表索引与页面查询方式对齐。
4. 避免在真实数据刚接入后，列表页和管理页就被低效分页与查询拖垮。

本轮设计不覆盖：

- 帖子详情页的最终大回复量渲染方案
- 全文搜索
- 复杂筛选器
- 归档页
- 缓存层和搜索引擎引入

## 设计结论

这一子项目采用以下方向：

- 公开帖子列表页采用游标分页。
- 列表默认按活跃时间倒序，即 `lastReplyAt` 优先排序。
- Admin 导入记录页只展示最近任务，按时间倒序加载更多。
- Admin 页第一版不做复杂筛选、全文搜索和页码分页。
- 页面查询方式与数据库索引一起设计，而不是先写 UI 再补性能。

## 为什么优先做这两类页面

你已经明确最担心的页面是：

- 帖子列表页
- Admin 导入记录页

这是合理的，因为：

- 帖子列表页会最先暴露排序、分页和索引问题。
- Admin 导入记录页会在同步 API 导入与旧库迁移开始后快速增长。

相比之下：

- 帖子详情页的性能问题虽然也重要，但更适合在真实回复量出现后再单独细化。

## 帖子列表页设计

### 读取目标

帖子列表页第一版的核心目标不是任意深历史快速跳页，而是稳定地读取最近活跃内容。

因此第一版策略明确为：

- 默认读取某个版面的最近活跃帖子
- 排序以 `lastReplyAt desc` 为主
- 使用游标分页而不是页码分页

### 为什么使用游标分页

游标分页优于传统页码分页的原因是：

- 当新帖子或新回复不断写入时，页码分页更容易出现翻页漂移
- 活跃排序下，页码越深，查询成本通常越高
- 游标分页更适合“加载更多最近内容”的产品体验

这与 `iwhisper` 的内容消费方式非常匹配。

### 排序规则

第一版按 `lastReplyAt` 倒序排序。

含义：

- 最近有新回复的帖子优先靠前
- 没有新回复的帖子也应有可回退的时间字段支撑排序稳定性

实现时需要保证排序键稳定，避免两个时间值相同时分页重复或跳项。

### 查询结构

第一版列表查询应当具备以下特征：

- 版面维度查询
- 固定排序方向
- 固定分页大小
- 通过游标拿下一页

第一版不追求：

- 自定义多排序模式
- 任意组合筛选
- 深分页跳转

## 帖子列表页索引要求

为了支撑上述查询方式，数据层至少需要：

- `threads(boardId, lastReplyAt)` 索引
- 在必要时补充稳定次级排序键

设计原则：

- 页面读取方式必须反推索引
- 索引要服务真实热路径，而不是为了理论完整性泛滥增加

## Admin 导入记录页设计

### 第一版目标

Admin 导入记录页第一版只需要解决一个问题：

让运营者稳定看到最近发生了什么导入与迁移任务。

因此第一版应聚焦：

- 最近任务
- 时间倒序
- 加载更多
- 轻量轮询刷新正在进行中的任务

### 不做的内容

第一版明确不做：

- 全文搜索
- 多条件筛选器
- 高级任务详情抽屉
- 页码分页
- 实时推送

这不是能力缺失，而是为了把读取成本控制在最能产生价值的地方。

### 数据来源

这一页应当主要展示：

- `imports`
- `importJobs`

可以按统一时间线视图呈现，也可以在实现阶段根据界面清晰度拆分成两个列表，但读取策略保持一致：

- 按时间倒序
- 只看最近任务
- 支持继续加载更旧记录

### 展示字段

第一版建议展示：

- 任务或导入类型
- 来源类型
- 状态
- 开始时间
- 最近更新时间
- 导入帖子数
- 导入回复数
- 跳过回复数
- 错误摘要

对旧库迁移任务，还应额外能看到：

- 当前游标摘要

## Admin 页刷新策略

第一版不做实时推送，而是采用轻量轮询。

原则：

- 只对最近任务区域轮询
- 只在存在运行中任务时轮询
- 停止运行后回到静态读取

这样做的好处是：

- 实现简单
- 服务端压力可控
- 足够满足当前阶段“看得见进度”的需求

## 统一读取策略

帖子列表页与 Admin 页虽然业务不同，但在读取策略上有共同原则：

- 优先服务最近数据
- 使用增量加载，而不是深页码跳转
- 让查询方式和索引方式一起设计

这会让第一版系统在数据量快速上涨后，仍能保持比较平稳的响应行为。

## 与前面几个子项目的关系

这个子项目依赖前面已经确定的设计：

- 真实数据会持续导入 `threads`
- 旧库迁移会持续生成 `importJobs` 与 `imports`
- Admin 页面本身会承担同步与迁移控制面职责

因此这里的优化不是孤立页面优化，而是对整个真实数据接入路线的读路径收束。

## 错误与降级思路

第一版至少应避免以下问题：

- 深翻页时查询越来越慢
- 运行中任务列表无限刷新整表
- Admin 最近任务页一次性捞取全量历史记录

如果真实数据量超出预期，优先采用的降级方向应是：

- 收窄单页返回数量
- 收紧最近任务窗口
- 优先保障最近活跃帖子与最近导入任务的读取稳定性

而不是第一时间引入复杂基础设施。

## 验证重点

这一子项目完成后，至少应验证：

1. 帖子列表页可以按 `lastReplyAt` 倒序稳定游标分页。
2. 新数据写入后，列表继续翻页不会明显漂移或重复。
3. 列表查询能命中预期索引。
4. Admin 导入记录页只读取最近任务，并支持继续加载更旧记录。
5. Admin 页在存在运行中任务时可以轻量轮询刷新，而不是频繁全量重载。
6. 导入与迁移记录增长后，页面仍能保持稳定响应。

## 非目标

这份设计明确不定义：

- 搜索引擎接入
- 全文搜索
- 复杂归档导航
- 回复超长详情页渲染优化
- 推送式任务订阅

这些应当在真实数据规模和使用模式更明确后再独立展开。
