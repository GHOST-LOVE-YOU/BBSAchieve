 # 首页板块固化与批量全量抓取设计

## 概述

这份设计用于取代同日较早那版“每板块单独全量任务”的方向。

当前新的目标是：

1. 先对北邮人首页板块做一次抓取。
2. 将抓到的全部板块结果固化进代码常量文件。
3. 在 `/admin/imports` 展示全部固化板块，包含 `IWhisper`。
4. 支持管理员一次勾选一个或多个板块，创建一条“批量全量抓取总任务”。
5. 总任务按代码中固化的板块顺序串行执行。
6. 某个板块失败时整条总任务立即失败并停止。
7. 恢复总任务时，从失败板块重新开始，之前成功的板块跳过。
8. 定时任务仍然只给代码里显式启用的板块生成，不与“全部固化板块”自动等同。

这份设计覆盖新的产品形态，因此后续实现不再以“每板块一条独立手动全量任务”为默认方案。

## 当前决策

你已经明确选择了以下方向：

- 首页板块抓取只需要做一次。
- 抓到的结果直接固化进代码常量文件。
- 后续靠手工维护，不保留自动再生成脚本。
- `/admin/imports` 使用“板块列表 + 多选框 + 一个开始全量抓取按钮”。
- 点击后创建一条总任务，而不是多条独立任务。
- 总任务按代码固化顺序串行跑多个板块。
- 某个板块失败就整条总任务失败。
- 恢复时从失败板块继续，之前成功的板块跳过。
- `IWhisper` 也出现在同一套板块多选列表里。
- 定时任务只给代码里显式启用的板块生成，板块列表和定时任务列表不是同一集合。

## 不覆盖的内容

本轮设计不覆盖：

- 再做一个“重新抓首页并自动改代码”的脚本。
- 在线增加、删除或修改板块清单。
- 在线调整定时任务是否启用、频率或窗口。
- 多实例分布式锁。
- 真正强中断一个已经发出的长抓取请求。
- 持久化任务队列或独立 worker 服务。

## 设计结论

本轮设计采用以下形态：

- BYR 首页板块抓取结果固化进代码常量。
- 固化板块清单是 `/admin/imports` 的唯一板块数据源。
- 页面支持多选板块，并通过一次提交创建一条批量全量抓取总任务。
- 总任务使用 `import_jobs` 承载，不新增第二张长任务表。
- 总任务的板块级进度保存在 `metadataJson` 中。
- 总任务一次只推进一个板块，并继续复用现有同步接口链路。
- 失败即停，恢复时从失败板块继续。
- 定时任务注册仍然由代码显式控制，不自动覆盖全部固化板块。

## 为什么不用“多条子任务”

这一轮有三条主要路线：

1. 一次多选，创建一条总任务，内部串行推进多个板块。
2. 一次多选，但底层仍创建多条独立板块任务。
3. 页面多选只是便捷入口，本质仍循环触发单板块任务。

本轮明确选择第 `1` 条路线。

原因：

- 你的目标是“一次选择多个板块，然后创建一条总任务”，而不是多个散落任务。
- “失败即停”和“从失败板块恢复”在总任务模型中语义最清楚。
- 管理后台更容易给出明确的当前板块、已完成板块和失败板块。

放弃后两条路线的原因是：

- 底层和 UI 语义会打架。
- 容易出现“父任务说失败、子任务还在跑”这类状态不一致。
- 恢复点和顺序控制会更混乱。

## 板块清单来源

### 首页抓取一次

先对北邮人首页板块抓取一次。

这一步的目的不是运行时动态发现，而是为了生成一份人工可维护的静态板块清单。

### 固化进代码

抓到的板块清单直接落进代码常量文件，例如：

- `apps/web/src/server/boardSync/boardCatalog.ts`

建议每个板块至少保存：

- `boardName`
- `boardSlug`
- `title`
- `description`
- `fullSyncEnabled`
- `scheduledSyncEnabled`
- `scheduledIntervalMinutes`
- `scheduledWindowMinutes`
- `fullSyncWindowMinutes`

其中：

- 是否显示在全量抓取多选列表，取决于是否在这份固化清单里存在
- 是否参与定时任务，则只看 `scheduledSyncEnabled`

### 与现有注册表的关系

当前已经存在的 `boardRegistry.ts` 更接近“少量人工板块配置”。

本轮设计建议将其升级为真正的“固化板块总目录”，并让：

- 全量抓取多选列表
- 定时任务派生
- 管理页展示文案

都从同一份目录派生，避免出现多份板块来源。

## `/admin/imports` 交互

### 页面入口

`/admin/imports` 页面顶部改为板块列表多选区。

每个板块显示：

- 复选框
- 板块名
- 必要时的简短说明

全部固化板块都应出现在这里，包括 `IWhisper`。

### 创建总任务

页面底部提供一个统一按钮，例如：

- `开始全量抓取`

点击后提交所选板块名数组，而不是某个单一板块名。

### 任务列表

页面下方不再是“旧库迁移任务”或“单板块任务”表，而是“批量全量抓取任务”列表。

每条任务至少展示：

- 任务状态
- 选中板块数
- 已完成板块数
- 当前板块
- 失败板块
- 累计导入帖子数
- 累计导入回复数
- 错误摘要
- 继续 / 停止操作

## 数据模型

### 继续复用 `import_jobs`

本轮继续复用 `import_jobs`，不新增第二张长任务表。

新增或统一使用：

- `jobType = byr_board_full_sync_batch`

建议：

- `sourceType = byr_sync_api`
- `sourceLabel = multi-board full sync`

### `metadataJson`

总任务的关键状态记录在 `metadataJson` 中。

建议至少包含：

- `selectedBoardNames`
- `orderedBoardNames`
- `completedBoardNames`
- `currentBoardName`
- `failedBoardName`
- `currentBoardIndex`
- `perBoardStats`

其中：

- `selectedBoardNames`
  - 管理员实际勾选的板块集合

- `orderedBoardNames`
  - 按代码固化顺序排序后的板块集合

- `completedBoardNames`
  - 已成功完成的板块

- `currentBoardName`
  - 当前正在尝试执行的板块

- `failedBoardName`
  - 本轮失败的板块

- `perBoardStats`
  - 每个板块各自的帖子/回复统计

### 顶层统计字段

`processedThreads`、`processedReplies` 继续作为总任务累计值保留在顶层字段中。

## 执行流程

### 创建任务

1. 管理员在 `/admin/imports` 勾选一个或多个板块。
2. 服务端校验这些板块都存在于固化清单中。
3. 按固化清单顺序重排所选板块。
4. 创建一条 `byr_board_full_sync_batch` 类型的 `import_jobs` 记录。
5. 将选中集合、排序集合和初始进度写入 `metadataJson`。
6. 启动后台总任务 runner。

### 板块推进

总任务 runner 每次只推进一个板块：

1. 读取 `currentBoardIndex` 和 `orderedBoardNames`。
2. 找到当前要执行的板块。
3. 尝试进入全局抓取节流器。
4. 如果没拿到节流器：
   - 整条任务记为 `paused`
   - `progressNote` 写明当前板块和等待原因
   - 本轮结束
5. 如果拿到节流器：
   - 调用现有同步导入链路执行当前板块全量抓取
   - 成功后更新 `completedBoardNames`、累计统计和 `perBoardStats`
   - 推进到下一个板块

### 失败即停

如果某个板块失败：

- 写入 `failedBoardName`
- 总任务标记为 `failed`
- 不再继续后续板块

### 恢复

点击“继续”时：

- 从 `failedBoardName` 对应位置重新开始
- `completedBoardNames` 中已有的板块全部跳过
- 不重跑之前成功的板块

## 与定时任务的关系

固化板块清单和定时任务列表不是同一集合。

本轮明确：

- 全量抓取多选列表 = 全部固化板块
- 定时任务列表 = 只包含 `scheduledSyncEnabled = true` 的板块

也就是说：

- 某板块可以允许手动全量抓取，但没有定时任务
- `IWhisper` 和其它板块是否进入定时任务，只看代码配置，不看它是否出现在多选列表

## 限速与共存

总任务仍与现有定时任务共用同一个全局抓取节流器。

因此：

- 总任务在某一板块抓取时，定时任务触发会直接 `skipped`
- 如果定时任务正在抓取，总任务在当前板块上会暂停

这保证：

- 任意时刻仍然只有一个真实抓取流程打主站

## 停止语义

总任务支持 `cancelled`。

含义：

- 之后不再继续下一板块
- 如果当前板块还没真正开始，直接停止在当前位置
- 如果当前板块抓取已经在进行中，则按当前架构尽量在安全点停止，不承诺强中断网络请求

## 测试重点

最少覆盖：

1. 固化板块清单可以派生多选列表和定时任务列表。
2. `/admin/imports` 正确展示全部固化板块的复选框。
3. 多选提交后只创建一条总任务。
4. 总任务按代码顺序推进，而不是按勾选顺序推进。
5. 某板块失败时整条任务立即失败。
6. 恢复时从失败板块继续，之前成功板块跳过。
7. 总任务和定时任务会因为全局抓取节流器而相互让行。

## 与旧设计的关系

这份设计取代同日较早那版“每板块一条独立手动全量任务”的方向。

如果后续继续实现，应以本 spec 为准，而不是继续围绕旧的单板块任务入口扩展。
