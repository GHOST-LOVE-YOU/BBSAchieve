# Web 端硬编码定时任务与窗口化同步设计

## 概述

这份设计聚焦当前一个独立且边界清晰的子项目：为 `apps/web` 增加代码硬编码的定时任务能力，并让第一条任务稳定同步 `IWhisper` 的最近内容到 Web 自己的数据库。

本轮设计覆盖以下目标：

1. 让 `web` 进程具备单实例常驻的定时任务执行能力。
2. 第一条任务固定为每 `20` 分钟同步一次 `IWhisper`，回看最近 `30` 分钟。
3. 保持任务定义、启停状态都硬编码在代码里，不放进数据库。
4. 让 `admin` 可见全部定时任务，并支持“立即执行一次”。
5. 保持现有“后端采集 -> Web 整理 -> 写入云端数据库”的职责边界。
6. 把后端同步 API 从“第一页前 N 条增量”升级为“按时间窗口翻页到越界为止”的同步入口。

本轮设计不覆盖：

- 多实例部署下的分布式锁或主实例选举。
- 停机超过时间窗口后的自动补偿策略。
- 在线修改任务定义、在线暂停任务或在线调整任务窗口。
- 复杂告警系统。
- 除 `IWhisper` 之外的第二条及后续定时任务实现细节。

其中“停机超过窗口后的补偿策略”被明确延期，本次不做设计决策，也不假装已经覆盖。

## 设计结论

这一子项目采用以下方向：

- 定时任务运行在 `web` 进程内，采用硬编码注册表。
- 任务定义和启停状态全部保留在代码里，数据库只保存执行历史。
- `admin` 只读任务定义与执行历史，不提供在线改配置能力。
- 手动“立即执行一次”和自动调度复用同一条 runner。
- `backend` 的同步 API 负责“按时间窗口翻页采集到哪里”，`web` 只负责“什么时候跑”和“把结果写进自己的库”。
- 第一条任务固定为：
  - `taskKey`: `iwhisper_recent_sync`
  - `board_name`: `IWhisper`
  - `interval_minutes`: `20`
  - `window_minutes`: `30`
  - `enabled`: `true`
- 本地环境通过显式开关关闭调度器；默认启用。

## 为什么采用 Web 进程内硬编码调度器

这次有 3 条可行路线：

1. `web` 进程内硬编码调度器，数据库只存执行历史。
2. `web` 进程内硬编码调度器，数据库再存任务状态。
3. 外部 cron 或平台定时器触发 Web API。

本轮选择第一条路线，原因是：

- 你已经明确偏好“代码硬编码”，不希望把任务定义或启停状态放进数据库。
- 当前部署前提是云端单实例长期常驻进程，这使进程内调度器成为可接受的最小方案。
- 后续任务虽然会增加，但数量有限，且大概率仍是少量板块、固定周期的组合。
- `admin` 只需要看到任务与执行结果，并不需要在线配置中心。

放弃另外两条路线的原因是：

- 如果把任务状态持久化到数据库，会引入“代码定义”和“持久化状态”两份事实源。
- 如果使用外部 cron，本质上是外部系统在调度，而不是 `web` 自己具备定时任务能力。

## 当前项目上下文

当前仓库已经有以下基础：

- `backend` 已具备登录态管理、版面抓取、帖子抓取与同步 API。
- `apps/web` 已具备管理员页面、Prisma 数据库和同步导入链路。
- `apps/web/app/admin/api/imports/byr-sync/route.ts` 已经串起：
  - 请求同步 API
  - 补原帖/补缺失楼层
  - `mapSyncPayload`
  - `importSyncBatch`

同时，当前 Web 端已经存在一种“应用进程内后台执行”的模式：

- 旧库迁移任务通过 API 触发后，在 `web` 进程中以异步后台方式继续运行。

这说明“任务在 Web 进程内运行”并不是全新模式，而是在现有模式上往前推进一步。

## 真实抓取结论

为了避免误判版面时间格式，本轮设计前对真实 `IWhisper` 版面做了实测抓取。

在 `2026-05-03` 抓到的第 `1` 页真实数据中，已明确观察到两类时间格式：

- 当天主题或当天最新回复显示为 `HH:MM:SS`
- 较早主题或较早最新回复显示为 `YYYY-MM-DD`

本轮实测没有观察到 `MM-DD` 这种格式，因此本次设计不把它写成既定事实。

继续抓第 `2` 页和第 `10` 页时，当前后端登录态解码逻辑出现了 `UnicodeDecodeError`，因此本轮只能确认：

- 真实线上至少存在 `HH:MM:SS`
- 真实线上至少存在 `YYYY-MM-DD`
- `MM-DD` 是否真实存在，后续应继续补充样本验证

## 整体职责边界

### Backend 负责什么

`backend` 负责：

- 确定时间窗口内应该继续抓哪些版面页。
- 解析版面列表里的时间字符串。
- 决定何时停止翻下一页。
- 对窗口内候选主题继续应用现有增量语义与补拉语义。
- 对 Web 返回尽量精简但完整的同步结果。

### Web 负责什么

`web` 负责：

- 何时调度任务。
- 是否启用调度器。
- 如何调用后端同步 API。
- 如何把返回结果整理并写入自己的 PostgreSQL。
- 如何记录每次任务的执行历史。
- 如何向 `admin` 展示任务和最近执行结果。

### Admin 负责什么

`admin` 负责：

- 展示代码里定义了哪些任务。
- 展示任务的调度间隔、回看窗口与启用状态。
- 展示最近执行历史。
- 提供“立即执行一次”入口。

`admin` 本轮不负责：

- 修改任务定义。
- 修改任务启停状态。
- 手工传入临时窗口或自定义参数。

## 任务定义模型

任务定义保存在 `apps/web` 的 server-only 注册表中，例如：

- `apps/web/src/server/scheduler/taskRegistry.ts`

第一版每条任务建议具备以下字段：

- `taskKey`
- `title`
- `description`
- `sourceType`
- `sourceLabel`
- `boardName`
- `intervalMinutes`
- `windowMinutes`
- `enabled`
- `runnerType`

第一条任务固定为：

- `taskKey`: `iwhisper_recent_sync`
- `title`: `IWhisper 最近内容同步`
- `sourceType`: `byr_sync_api`
- `sourceLabel`: `IWhisper recent sync`
- `boardName`: `IWhisper`
- `intervalMinutes`: `20`
- `windowMinutes`: `30`
- `enabled`: `true`
- `runnerType`: `byr_sync_recent_window`

后续新增日更或周更任务时，只新增新的代码定义，不改这一层的设计原则。

## 调度器设计

### 启动入口

调度器建议通过 `apps/web/instrumentation.ts` 的 `register()` 启动。

原因：

- 它适合作为 Web server instance 启动时的统一入口。
- 不依赖某个页面先被访问。
- 比把调度器挂在页面模块或路由模块里更清晰。

### 启停开关

建议至少提供两个环境变量：

- `WEB_SCHEDULER_ENABLED=true`
- `WEB_SCHEDULER_RUN_ON_BOOT=true`

语义如下：

- `WEB_SCHEDULER_ENABLED=false`：
  - 整个调度器不启动。
  - 适合本地开发环境关闭定时任务。
- `WEB_SCHEDULER_RUN_ON_BOOT=true`：
  - 进程启动后先立即执行一次，再进入正常调度周期。

本轮设计采用“默认开启”的原则，但本地环境可显式关闭。

### 单例与重入保护

调度器在进程内应保持单例，只初始化一次。

每个任务还需要自己的运行中标记：

- 如果某条任务正在执行，本轮触发不应再并发启动第二次。
- 跳过也应记一条执行历史，状态记为 `skipped`，原因写明 `previous run still active`。

本轮不做：

- 分布式锁
- 跨实例抢占
- 任务优先级调度

因为当前前提是单实例常驻。

## 任务执行数据流

每条任务的自动执行与手动执行都走同一条 runner，流程固定如下：

1. 根据 `taskKey` 读取代码注册表中的任务定义。
2. 写入一条“开始执行”的历史记录。
3. 请求后端同步 API，并携带：
   - `board_name`
   - `window_minutes`
4. 获取后端已按窗口翻页和增量语义整理后的 payload。
5. 复用现有 Web 导入链路：
   - 视情况补原帖或补缺失楼层
   - `mapSyncPayload`
   - `importSyncBatch`
6. 更新这次任务历史为：
   - `succeeded`
   - `failed`
   - 或 `skipped`
7. 记录统计信息与错误摘要。

手动“立即执行一次”与自动调度的唯一区别只在于触发来源：

- `triggerSource=scheduled`
- `triggerSource=manual`

除此之外，执行配置完全一致，不接受额外参数，不走旁路逻辑。

## 执行历史模型

本轮建议为定时任务新增独立的执行历史表，而不是复用 `imports` 或 `importJobs`。

原因：

- `imports` 的语义是一次导入会话。
- `importJobs` 的语义偏向长任务控制与进度。
- 定时任务需要的核心是“固定代码任务的每次运行记录”。

如果继续复用现有表，随着定时任务种类增加，语义会越来越混。

### 建议表结构

建议新增一张例如 `scheduled_task_runs` 的表，至少包含：

- `id`
- `taskKey`
- `taskTitle`
- `triggerSource`
- `status`
- `startedAt`
- `finishedAt`
- `durationMs`
- `intervalMinutes`
- `windowMinutes`
- `boardName`
- `importedThreads`
- `importedReplies`
- `skippedReason`
- `errorMessage`
- `metadataJson`
- `createdAt`
- `updatedAt`

建议的状态至少包括：

- `running`
- `succeeded`
- `failed`
- `skipped`

### 为什么数据库只存执行历史

数据库本轮只存执行历史，不存任务状态，有三个好处：

- 代码定义是唯一事实源。
- `admin` 展示清晰，不会出现“代码启用但数据库禁用”的冲突。
- 后续如果决定引入在线控制面，也可以在下一轮独立设计，不污染当前最小方案。

## Admin 展示设计

建议在 `admin` 下新增独立的定时任务页面，例如：

- `/admin/scheduled-tasks`

不建议把它继续塞进现有“导入导出”页面的按钮区块里。

原因：

- 它的主视角是“任务清单 + 最近执行结果”。
- 它不是一次性的操作入口，而是持续存在的控制面。

### 页面展示内容

每条任务建议展示：

- 任务名称
- 说明
- 板块名
- 调度间隔
- 回看窗口
- 代码启用状态
- 最近一次状态
- 最近一次开始时间
- 最近一次结束时间
- 最近一次导入帖子数
- 最近一次导入回复数
- 最近一次错误摘要
- “立即执行一次”按钮

### 手动执行接口

建议新增一个管理接口，例如：

- `POST /admin/api/scheduled-tasks/[taskKey]/run`

接口语义：

- 只接受 `taskKey`
- 校验任务是否存在且在代码中启用
- 触发同一个 runner
- 返回本次运行的历史记录 ID 或任务键

本轮不支持：

- 自定义窗口
- 自定义版面
- 执行时覆盖任务配置

## 后端同步 API 改造

### 当前问题

当前 `backend` 的同步 API 本质上还是：

- 抓 `IWhisper` 第 `1` 页
- 取前 `limit` 个主题
- 对这些主题按 reply count 决定是否补抓帖子页

这与“每 `20` 分钟同步一次，回看最近 `30` 分钟”的目标并不匹配。

### 改造目标

本轮要把同步 API 升级成：

- 按时间窗口驱动的版面翻页同步入口

建议把 `GET /api/sync/updates` 改成显式支持：

- `board_name`
- `window_minutes`

第一版任务固定传：

- `board_name=IWhisper`
- `window_minutes=30`

### 翻页与停止规则

版面抓取从第 `1` 页开始顺序翻页。

对于每个非置顶主题，后端先计算“判定时间”：

- 优先使用 `最新回复时间`
- 如果没有回复，则退回 `发帖时间`

然后与“当前时间减去窗口分钟数”的截止时间比较。

停止规则如下：

- 只要当前页出现窗口外主题，就把这一页中仍在窗口内的主题保留。
- 当前页处理完后停止继续翻下一页。

这样可以保证：

- 按版面最近活动顺序采集
- 不漏掉边界页中仍在窗口内的主题
- 不继续翻明显越界的后续页面

### 时间解析规则

基于本轮实测，第一版至少支持以下两类时间字符串：

- `HH:MM:SS`
- `YYYY-MM-DD`

解析规则如下：

- `HH:MM:SS`
  - 解释为论坛时区当天的具体时间
- `YYYY-MM-DD`
  - 解释为该日期的 `23:59:59`

之所以把日期补成 `23:59:59`，是为了遵守本项目在同步上的偏保守策略：

- 宁可多抓一点，也不要因为时间颗粒不足而漏掉可能仍在窗口边界附近的主题

本轮不把 `MM-DD` 写成既定规则，但解析器实现应预留可扩展结构，方便后续补入。

### 增量语义仍然保留

时间窗口只负责决定“哪些主题值得被纳入本次同步候选集”。

在此之后，后端仍应继续复用现有增量语义：

- reply count 与缓存对比
- 是否需要抓帖子页
- 是否需要补原帖或补缺失楼层

这样可以同时满足：

- 窗口同步减少不必要的数据传输
- 现有缓存与增量逻辑继续发挥价值

### 返回结构

返回结构应尽量贴近现有 payload，降低 Web 侧改动成本。

建议保留：

- `board_name`
- `threads[]`

可新增少量元信息：

- `window_minutes`
- `scanned_pages`
- `cutoff_at`

但不建议为了这次定时任务重写整个返回结构。

## Web 导入链路复用

这次调度器不重新发明导入逻辑，而是复用现有 Web 端的同步导入链路。

因此：

- `fetchSyncUpdates` 需要支持带查询参数请求新的同步 API
- 现有补原帖、补缺楼、`mapSyncPayload`、`importSyncBatch` 继续保留
- 定时任务 runner 只是在其外部多包一层“任务编排 + 执行历史”

这样做的好处是：

- 降低本轮实现风险
- 复用已有测试与数据写入能力
- 避免调度器和导入器形成两套写库逻辑

## 失败处理与可见性

本轮失败处理保持简单透明：

- 后端同步 API 失败，记失败历史。
- Web 写库失败，记失败历史。
- 时间解析失败，记失败历史，并包含错误摘要。
- 任务未注册或未启用，手动触发接口直接返回错误。

调度器本身不应因为某次执行失败而停掉整个循环。

也就是说：

- 单次失败只影响当次记录
- 后续周期继续正常调度

本轮不引入：

- 邮件告警
- 短信告警
- Webhook 告警

先保证 `admin` 页面能看见最近失败即可。

## 与后续多任务扩展的关系

虽然本轮只有一条任务，但设计必须为后续任务扩展留出稳定边界。

后续你提到的这类任务都应能复用同一套机制：

- 某板块每天执行一次，回看最近两天
- 某板块每周执行一次，回看最近八天

本轮为此预留的关键点是：

- 任务注册表而不是单个写死函数
- `board_name + interval_minutes + window_minutes` 的任务定义方式
- 独立的执行历史表
- admin 的任务列表展示

## 测试重点

本轮实现完成后，至少应验证：

1. Web 调度器在启用时能启动，在禁用时完全不启动。
2. `instrumentation.ts` 只会触发一次调度器初始化。
3. 同一任务执行中再次触发时会被跳过，并记录 `skipped` 历史。
4. 手动“立即执行一次”和自动调度会走同一条 runner。
5. 后端同步 API 能按 `board_name + window_minutes` 正常返回窗口内候选主题。
6. 后端会在边界页处理完后停止继续翻页。
7. `HH:MM:SS` 能按当天时间解析。
8. `YYYY-MM-DD` 能按 `23:59:59` 解析。
9. Web 侧可以复用现有导入链路把结果写进数据库。
10. Admin 页面能展示全部代码任务和最近执行结果。

## 风险与延期项

本轮明确接受以下风险或延期项：

- 单实例前提未来变化后，需要额外补分布式锁方案。
- 停机超过时间窗口后的补偿策略暂不处理。
- `MM-DD` 时间格式尚未在真实抓取中确认，不纳入本轮既定规则。
- 当前后端抓第 `2` 页及更深页时暴露出的登录态解码问题，需要在实现阶段顺手修掉，否则会影响真正的窗口翻页能力。

这些不是本轮设计遗漏，而是有意识地压缩范围，优先先跑通“单实例 Web 调度器 + 窗口化后端同步 + Admin 可见”的最小闭环。
