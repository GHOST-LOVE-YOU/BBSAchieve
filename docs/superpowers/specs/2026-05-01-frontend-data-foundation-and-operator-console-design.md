# 前端数据底座与双端第一阶段设计

## 概述

这份设计聚焦前端第一阶段子项目，目标是先建立可长期演进的前端数据底座，并同时为 Web 与移动端提供正确的产品边界。

这份设计解决的问题是：

1. 在暂不接入真实后端的前提下，先让前端具备自己的论坛内容存储与维护能力。
2. 让 Web 端既能作为真实用户访问的论坛入口，也能承载受权限保护的管理员内容运营界面。
3. 让移动端从一开始就面向真实用户，而不是复用管理员思路。
4. 为后续接入真实后端、正式认证、通知与认领功能预留稳定边界。

这份设计不覆盖最终视觉设计系统、不覆盖正式认证闭环、不覆盖完整通知产品、不覆盖真实同步后端接入，也不定义最终云端数据库方案。

## 设计结论

前端第一阶段采用以下方向：

- Web 使用 Next.js。
- 移动端使用 Expo（React Native）。
- 共享层复用 API 抽象、部分状态管理、领域模型与基础组件接口。
- UI 暂不做正式视觉设计，只保留基础 `<div>`、`<View>` 和最小组件复用边界。
- 本地数据库直接存“沙盒论坛内容”，不同步保留单独的“北邮人镜像层”。
- Web 同时服务真实用户论坛入口与管理员运营入口，管理员区挂在 `/admin/...`。
- 移动端只面向真实用户，提供最小产品闭环。

## 产品边界

### 第一阶段的核心目标

第一阶段不是做一个接口调试壳，而是做前端自己的论坛内容底座。

优先目标是：

1. 让前端能存储完整论坛雏形所需的核心内容实体。
2. 让 Web 管理员区能够维护这些内容，包括导入假数据、创建机器人、创建帖子与回复、查看时间线。
3. 让 Web 公开区与移动端都能以真实用户视角浏览版面、帖子与回复。
4. 为未来接入后端同步、通知、认领、认证与云端数据库打下结构基础。

### 非目标

这一阶段明确不做：

- 最终视觉设计系统
- 正式登录与鉴权实现
- 完整通知中心
- 机器人自动发帖编排
- 真实同步后端接入
- 最终云端数据库落地
- 完整管理员权限系统

## 为什么需要本地数据库

本地数据库不是为了替代未来云端数据库，而是为了在当前阶段承担客户端侧业务存储职责。

它至少需要解决以下问题：

1. 持久化论坛内容  
   版面、帖子、回复、作者、草稿、导入记录、时间线等不能只留在内存中。

2. 支撑完整 CRUD  
   前端不仅展示内容，还要创建、编辑、删除帖子、回复、机器人用户和相关运营记录。

3. 支撑移动端与弱网场景  
   尤其是移动端，需要在网络不稳定或后端未接入时仍能正常读取已有内容。

4. 承接“前端先行、后端后接”的开发顺序  
   假数据导入、手工维护和产品验证都依赖本地持久化。

5. 提供双端共享的数据语义  
   Web 与 Expo 不一定共用同一种数据库实现，但需要共用同一套论坛内容模型与仓储语义。

未来仍然需要云端数据库。云端数据库承担长期一致的全局事实源，本地数据库承担设备侧可操作副本、缓存与离线工作区。两者不是替代关系，而是配合关系。

## 整体架构与工作区边界

前端建议采用 monorepo 工作区结构，按“应用层 / 共享业务层 / 平台基础层”分层。

### 应用层

- `apps/web`
  - Next.js 应用
  - 默认面向真实用户提供论坛访问入口
  - 同时包含受权限保护的管理员区 `/admin/...`

- `apps/mobile`
  - Expo 应用
  - 仅面向真实用户
  - 不承载管理员运营职责

### 共享业务层

- `packages/domain`
  - 论坛实体
  - 值对象
  - 领域规则
  - 仓储接口
  - 错误模型

- `packages/state`
  - 可跨端复用的查询状态
  - 部分表单状态
  - 操作编排逻辑
  - 读写用例封装

- `packages/api`
  - 未来对接真实后端的 API client
  - DTO 与领域模型转换
  - 同步入口抽象

### 平台基础层

- `packages/ui-base`
  - 最小基础组件边界
  - 布局容器、列表壳、详情壳、表单壳、按钮壳、输入壳
  - 不定义最终视觉风格

- `packages/config`
  - tsconfig、lint、环境变量约定、共享构建配置

- `packages/test-utils`
  - 测试假数据构造器
  - 仓储 mock
  - 跨端用例测试辅助

### UI 暂缓策略

这一阶段明确不做 UI 设计系统，只保留：

- 结构占位
- 基础组件接口
- 页面骨架
- 跨端可复用的最小布局原语

这保证本阶段精力集中在数据底座与产品边界，而不会提前固化视觉决策。

## 数据模型与实体边界

### 总体原则

本地数据库直接存“沙盒论坛内容”，不单独维护“北邮人镜像内容层”。

无论内容来自：

- 假数据导入
- 管理员手工录入
- 未来同步接入

最终都进入同一套论坛实体模型。

同步来源如果未来需要保留，只作为导入记录或同步记录的一部分存在，而不是单独的内容层。

### 身份模型

身份模型采用“共享基础用户表 + 角色扩展表”。

原因是：

- 真实用户未来需要接第三方认证，例如 Kinde。
- 机器人用户不登录、不需要 dashboard，只作为论坛中的发帖主体存在。
- 未来真实用户可能绑定机器人收件箱通知，因此两者必须能自然建立关联。

建议实体如下：

- `users`
  - `id`
  - `username`
  - `display_name`
  - `user_type`（`human` / `bot`）
  - `avatar_url`
  - `bio`
  - `status`
  - `created_at`
  - `updated_at`

- `human_profiles`
  - `user_id`
  - `auth_provider`
  - `auth_subject`
  - `email`
  - `profile_status`

- `bot_profiles`
  - `user_id`
  - `mailbox_key`
  - `source_label`
  - `can_post`
  - `persona_summary`
  - `profile_status`

- `user_bot_bindings`
  - `id`
  - `human_user_id`
  - `bot_user_id`
  - `binding_status`
  - `created_at`

这个结构保证帖子、回复、通知等业务表只需要统一引用 `users.id`，同时又能为真人和机器人保留各自独立的扩展字段。

### 论坛内容模型

建议第一阶段落以下核心内容实体：

- `boards`
  - `id`
  - `slug`
  - `name`
  - `description`
  - `visibility`
  - `created_at`

- `threads`
  - `id`
  - `board_id`
  - `author_user_id`
  - `title`
  - `body`
  - `thread_status`
  - `created_at`
  - `updated_at`
  - `published_at`

- `replies`
  - `id`
  - `thread_id`
  - `author_user_id`
  - `body`
  - `reply_index`
  - `reply_status`
  - `created_at`
  - `updated_at`
  - `published_at`

几个关键约束：

- `author_user_id` 统一指向 `users.id`
- 帖子与回复使用论坛内部主键
- 外部来源 ID 不作为核心主键
- 用户、帖子、回复优先使用软删除而不是硬删除

### 内容运营辅助模型

第一阶段还需要最小的运营辅助实体：

- `imports`
  - 记录一次导入会话

- `import_items`
  - 记录单项导入结果、错误定位与关联实体

- `content_events` 或等价时间线表
  - 记录创建、编辑、删除、导入等内容事件

- `sync_sessions`
  - 记录未来同步写入的会话边界

- `sync_targets`
  - 记录未来同步目标与补拉入口关联信息

这里的同步辅助实体只承担“来源可追踪”的职责，不在第一阶段发展成复杂调度系统。

## 仓储接口与数据库适配边界

### 共享什么

共享层共享的是：

- schema 语义
- 领域实体
- 仓储接口
- 错误模型
- 查询参数
- 写入规则

而不是共享同一套数据库实现。

### 不共享什么

Web 与移动端不要求：

- 使用同一个数据库库名
- 使用同一份迁移脚本
- 使用完全相同的存储引擎

这样可以避免在第一阶段把精力过早投入到跨平台数据库工具链统一上。

### 仓储接口

`packages/domain` 中建议定义以下仓储接口：

- `UserRepository`
- `BoardRepository`
- `ThreadRepository`
- `ReplyRepository`
- `ImportRepository`
- `TimelineRepository`
- `SyncSessionRepository`

这些接口表达的是前端需要的业务能力，而不是底层 SQL 或平台存储细节。

### 平台适配

- `apps/web`
  - 实现 Web 侧 repository adapter

- `apps/mobile`
  - 实现移动端 repository adapter

两边都遵守相同仓储接口，但具体存储方案可不同。第一阶段只要求这些实现满足：

- 能表达关系型或等价关系模型
- 支持索引、过滤、排序、分页
- 支持事务或等价批处理
- 支持导入导出
- 支持软删除
- 支持审计时间戳

### 第一批必须覆盖的仓储能力

1. 用户与身份
   - 按用户名查找用户
   - 创建机器人用户
   - 创建真实用户占位记录
   - 查询用户详情
   - 停用或软删除用户

2. 论坛内容
   - 列出版面
   - 按版面分页列帖子
   - 读取帖子详情
   - 读取回复流
   - 创建帖子
   - 编辑帖子
   - 创建回复
   - 编辑回复
   - 软删除帖子或回复

3. 运营辅助
   - 写入时间线事件
   - 创建导入会话
   - 批量导入用户、帖子、回复
   - 导出指定内容范围
   - 查询导入结果与失败项

4. 轻量同步辅助
   - 记录一次同步会话
   - 记录补拉目标
   - 关联同步写入与内容写入来源

## 双端页面边界

### Web：默认论坛 + 管理员区

Web 是同一个 Next.js 应用，默认路由面向真实用户，管理员能力挂在 `/admin/...` 下。

公开区建议承担：

- 首页
- 版面列表
- 帖子列表
- 帖子详情与回复流
- 未来登录与用户入口占位

管理员区建议承担：

- `/admin`
  - 内容总览

- `/admin/users`
  - 用户管理
  - 创建机器人
  - 维护真实用户占位记录

- `/admin/boards`
  - 版面与帖子管理

- `/admin/threads/[threadId]`
  - 帖子详情、回复管理、内容编辑

- `/admin/imports`
  - 假数据导入导出
  - 导入结果与失败项查看

- `/admin/activity`
  - 内容时间线

这样 Web 既能支持真实用户使用，也能在后续认证接入后，为管理员开放受限入口。

### Mobile：真实用户最小闭环

移动端第一阶段只面向真实用户，提供最小产品闭环：

- 版面列表
- 帖子列表
- 帖子详情
- 回复流
- 轻量通知/绑定入口占位

这里的通知/绑定入口不要求第一阶段完成完整功能，但需要在信息架构上为未来“真实用户绑定机器人收件箱通知”留出入口。

移动端不提供管理员入口。

## 关键操作流

### Web 公开区阅读流

真实用户在 Web 公开区可以：

1. 浏览版面列表
2. 查看版面下的帖子列表
3. 进入帖子详情页查看主帖与回复
4. 查看作者基础信息与最小时间线展示

这一阶段可以先保留登录相关入口占位，但不要求完成正式认证。

### Web 管理区内容运营流

管理员在 `/admin` 区域内可以：

1. 查看内容总览
2. 创建或维护机器人用户
3. 创建或维护真实用户占位记录
4. 创建、编辑、软删除帖子
5. 创建、编辑、软删除回复
6. 导入假数据
7. 导出内容
8. 查看内容时间线与导入结果

这里最关键的业务流程是：

- 当创建帖子或回复时，如果目标机器人用户不存在，系统允许先创建 `users + bot_profiles`，再继续内容写入。

### Mobile 真实用户流

真实用户在移动端可以：

1. 浏览版面与帖子
2. 阅读帖子与回复
3. 进入轻量通知/绑定入口占位

第一阶段不要求移动端具备管理员式内容编辑能力。

### 内容写入共享规则

不论内容来自：

- 管理员手工录入
- 假数据导入
- 未来同步接入

最终都遵循同一套写入规则：

1. 先确认作者用户存在
2. 若作者为机器人且不存在，则先创建机器人用户
3. 写入帖子或回复内容
4. 写入对应时间线事件
5. 必要时写入导入会话或同步会话关联

## 导入导出设计

导入导出只在 Web 管理区提供。

### 导入

第一阶段优先支持结构化 JSON 导入。

最小导入对象包括：

- 用户
- 版面
- 帖子
- 回复

导入时按以下顺序执行：

1. 用户
2. 版面
3. 帖子
4. 回复

导入需要支持两种模式：

- 严格模式
  - 遇到关键错误即中断

- 尽力模式
  - 跳过失败项并输出完整报告

对机器人作者，导入流程允许自动补建机器人用户。

### 导出

第一阶段导出至少支持：

- 单版面导出
- 单帖子导出
- 按时间范围导出

导出结果应附带基础元信息，例如：

- 导出时间
- 数据版本
- 记录数量

## 错误处理与状态边界

第一阶段至少需要显式处理以下错误：

1. 作者不存在且未允许自动补建
2. 创建机器人时用户名冲突
3. 编辑或删除已软删除实体
4. 导入关键字段缺失
5. 导入过程中引用关系缺失
6. 管理区权限不足或未授权

系统不应使用静默失败或空成功状态掩盖这些问题。

移动端的通知/绑定入口如果尚未实现完整逻辑，应返回清晰的“待接入”状态，而不是空白或不可解释的界面。

## 时间线与审计

第一阶段需要提供最小可用的内容时间线。

时间线至少记录：

- 实体类型
- 实体 ID
- 动作类型
- 操作者类型
- 发生时间
- 可选备注

第一阶段不强求完整字段 diff，但必须保证未来能继续扩展审计能力。

## 测试重点

这轮测试重点应放在数据与流程，而不是视觉。

最少覆盖：

1. 创建机器人用户后成功创建帖子
2. 机器人不存在时自动补建并成功发帖
3. 帖子与回复 CRUD 正确写入时间线
4. 导入用户、帖子、回复的成功路径
5. 导入失败项被正确记录
6. 软删除后列表与详情查询行为符合预期
7. Web 公开区与 `/admin` 区职责分离
8. Mobile 最小阅读流可读取共享内容模型

共享层测试优先级高于页面测试，因为它决定后续 Web 与移动端能否稳定共用同一套业务语言。

## 与后续阶段的关系

这份设计是后续几个方向的前置基础：

1. 接入真实同步后端
2. 接入第三方认证，例如 Kinde
3. 建立通知与机器人收件箱绑定流程
4. 引入云端数据库作为长期事实源
5. 扩展真实用户产品能力

因此，这一阶段最重要的不是把所有功能都做完，而是确保：

- 数据模型不跑偏
- 双端边界不跑偏
- Web 真实用户区与管理员区不混淆
- 机器人与真实用户的身份关系被正确表达

## 实施建议

实施时应优先遵循以下顺序：

1. 建立 monorepo 工作区与共享包边界
2. 定义领域实体与仓储接口
3. 实现 Web 与移动端的最小数据库适配骨架
4. 跑通 Web 管理区的用户、帖子、回复 CRUD
5. 跑通导入导出与时间线
6. 补齐 Web 公开区与移动端最小阅读流
7. 最后再为未来认证、同步与通知预留接口

这个顺序能让前端尽早拥有自己的内容底座，并避免一开始就被视觉设计、认证接入或后端依赖阻塞。
