# 内容订阅通知设计

## 背景

原规划允许真实用户绑定机器人收件箱来接收通知，同时未来可能直接绑定或认领某个镜像帖子、镜像回复。重新审视后，这个模型会把“机器人身份”和“具体内容通知范围”混在一起。

机器人 ID 可能对应多个真实源站用户。如果真实用户绑定机器人，就会自然出现某些帖子要通知、某些帖子不要通知、某些回复要通知、某些回复不要通知的例外规则。这个方向会让通知逻辑过早复杂化。

## 设计结论

当前规划取消“绑定机器人”作为通知范围。

真实用户只能订阅具体内容：

- 订阅某个镜像帖子。
- 订阅某条镜像回复。

订阅只表达“接收通知”，不表达“认领身份”，也不表示该源站内容属于某个真实登录用户。

机器人仍然是论坛的一等参与者，但它在通知产品里只承担内容发现入口：

- 用户可以搜索机器人用户，但搜索范围只允许机器人用户。
- 用户可以进入机器人详情页。
- 机器人详情页展示最近发帖和最近回复。
- 用户可以从机器人详情页快速订阅具体帖子或回复。

## 数据模型

后续数据库以 `content_subscriptions` 表表达通知订阅关系。

核心字段：

- `humanUserId`：订阅者。
- `targetType`：订阅目标类型，支持 `thread` 和 `reply`。
- `threadId`：订阅帖子时填写。
- `replyId`：订阅回复时填写。
- `subscriptionStatus`：支持 `active`、`muted`、`revoked`。

约束：

- `threadId` 与 `replyId` 二选一。
- 同一用户不能重复订阅同一个帖子。
- 同一用户不能重复订阅同一条回复。
- 删除帖子或回复时，对应订阅可以级联删除。

## 产品入口

第一阶段先补齐发现能力：

1. 搜索回复。
2. 搜索机器人用户。
3. 机器人详情页展示最近发帖和最近回复。
4. 在帖子详情页、回复项和机器人详情页内容列表中提供订阅入口。

不提供绑定机器人入口，也不设计机器人绑定状态。

## 边界

本设计不实现通知投递、通知中心完整 UI、搜索接口和机器人详情页，只先修正项目规划、数据库边界与占位文案。
