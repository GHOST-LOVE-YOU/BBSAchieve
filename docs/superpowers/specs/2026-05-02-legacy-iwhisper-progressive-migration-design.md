# 旧库 iwhisper 渐进迁移设计

## 概述

这份设计聚焦第二个子项目：把旧 PostgreSQL 中的 `iwhisper` 历史内容逐步迁入新 Prisma 数据库，并支持边迁移边可用。

本轮设计覆盖以下目标：

1. 只迁移旧库中的 `iwhisper` 内容。
2. 支持边迁移边可用，而不是全量迁完再切换。
3. 明确迁移任务以帖子为最小恢复单位。
4. 让旧库迁移与同步 API 导入复用同一套内部标准导入 DTO 与 Prisma 写库服务。
5. 为 Admin 提供后台迁移任务入口、进度可见性和恢复能力。

本轮设计不覆盖：

- 旧库全量迁移到其他板块的策略。
- 多 worker 并发迁移。
- 回复级断点续传。
- 海量数据导入时的最终吞吐优化。
- 独立任务队列系统或独立 worker 拆分。

## 设计结论

旧库迁移采用以下方向：

- 入口放在 Web `/admin`。
- 迁移任务运行在 Web 服务端同进程后台任务中。
- 第一版只迁移 `iwhisper`。
- 迁移顺序采用“先新后旧”。
- 以“帖子”为最小恢复单位。
- 旧库作者统一按 `users.username` 归并，并落成 `bot` 用户。
- 旧库读取结果先适配为内部标准导入 DTO，再复用统一 Prisma 写库服务。
- 长任务状态单独记录在 `importJobs` 表中，单次导入结果继续记在 `imports`。

## 为什么采用边迁移边可用

你已经确认更希望“先导入一部分就能展示”。

因此迁移系统不应设计为：

- 一次性全量迁完再开放展示。
- 必须长时间占用人工关注才能继续。
- 失败后只能从头重跑。

而应设计为：

- 最近更有价值的帖子优先进入新库。
- 一批一批推进历史范围。
- 任意时刻中断后都能继续。
- 迁移一开始，公开阅读页就能直接受益。

## 迁移范围

第一版只处理旧库中的 `iwhisper`。

理由：

- 当前最需要的是让 `iwhisper` 的真实内容可展示。
- 旧库体量很大，先把板块范围收窄更稳妥。
- 先验证一套成熟的适配层、DTO 和写库流程，再考虑扩板块。

## 与同步 API 导入的关系

旧库迁移不应自己定义另一套写入语义。

本轮设计明确采用统一导入边界：

- Python 同步 API 返回结果
- 旧库查询结果

两者都先转换成同一个内部标准导入 DTO。

最终只有统一 Prisma 写库服务负责：

- 解析 DTO
- 幂等写入内容表
- 维护统计
- 生成导入记录

这样旧库迁移与同步 API 导入的主要差别只保留在“取数适配层”，而不是“写入目标完全不同”。

## 迁移顺序

迁移顺序采用“先新后旧”。

含义如下：

- 先迁最近的帖子。
- 最近活跃内容应更早进入新库。
- 历史更旧的内容按批次逐步补齐。

这样做的好处是：

- 公开阅读页会优先得到更有价值的内容。
- 更快验证真实用户会看到的热路径。
- 迁移过程中的产品收益更早出现。

## 最小恢复单位

第一版以“帖子”为最小恢复单位。

这意味着：

- 一次迁移任务处理一批旧库帖子。
- 每个帖子连同其回复一起转换并导入。
- 某个帖子导入成功后，推进迁移游标。
- 失败时从下一个未完成帖子继续，而不是从回复级别恢复。

这样做的原因是：

- 恢复语义已经足够稳定。
- 可以明显降低实现复杂度。
- 能满足“边迁移边可用”的目标，不必为了回复级断点引入过重状态管理。

## 用户归并规则

旧库迁移的作者归并规则如下：

- 旧库作者按 `users.username` 对齐。
- 迁移导入的作者统一建模为 `bot` 用户。
- 如果同名机器人用户已存在，则复用现有记录。
- 不在本轮设计中引入旧库专用用户映射表。

这个规则与同步 API 导入保持一致，能让历史数据和实时同步数据共享同一套作者模型。

## 后台任务运行模式

迁移任务运行在 Web 服务端同进程后台任务中。

第一版不引入：

- 独立 worker 服务
- 消息队列
- 分布式任务调度

原因：

- 当前最重要的是跑通迁移链路，而不是先建设复杂基础设施。
- 统一放在 Web 服务端内，最方便直接复用 Prisma、导入 DTO 与 Admin 控制面。
- 只要任务状态持久化到数据库，进程内任务模式已经足够支撑第一版恢复与可见性需求。

## 数据表设计

### `importJobs`

为长任务单独增加 `importJobs` 表。

建议字段：

- `id`
- `jobType`
- `sourceType`
- `sourceLabel`
- `status`
- `cursorThreadKey`
- `lastProcessedAt`
- `startedAt`
- `finishedAt`
- `processedThreads`
- `processedReplies`
- `skippedThreads`
- `skippedReplies`
- `errorMessage`
- `progressNote`
- `metadataJson`
- `createdAt`
- `updatedAt`

字段语义：

- `jobType`
  - 第一版至少支持 `legacy_iwhisper_migration`

- `status`
  - 至少支持 `pending`、`running`、`paused`、`failed`、`succeeded`、`cancelled`

- `cursorThreadKey`
  - 表示当前已经完成到哪个帖子
  - 不直接暴露旧库私有语义
  - 应尽量与内部标准导入 DTO 中的线程标识规则对齐

- `metadataJson`
  - 保存批大小、排序方向、恢复参数、旧库查询摘要等辅助信息

### `imports`

`imports` 继续保留，职责不变：

- 记录单次导入会话的结果摘要
- 记录成功、失败、部分成功的统计信息
- 为 Admin 提供近期开销与结果可见性

旧库迁移中，每一批帖子导入仍然可以写入 `imports`，但长任务生命周期本身由 `importJobs` 承接。

## 内部标准导入 DTO

本轮设计明确要求定义一个内部标准导入 DTO。

原则：

- 不直接把 Python 同步 API 当前响应结构当成数据库写入协议。
- 不直接把旧库私有字段结构传给 Prisma 写库层。
- 两边都通过适配层转换成统一内部结构。

这个内部 DTO 至少要能表达：

- 版面信息
- 线程来源标识
- 线程作者
- 线程标题与正文
- 回复列表
- 回复作者
- 回复楼层号
- 回复发布时间
- 一次导入的来源类型与来源标签

这样后续旧库迁移和同步 API 导入就能共享：

- 内容去重规则
- 作者落库规则
- 错误处理规则
- 导入统计规则

## 迁移流程

第一版迁移流程如下：

1. Admin 页面点击“从旧库导入 iwhisper”。
2. Web 服务端创建一条 `importJobs` 记录。
3. 后台任务开始读取旧库中的最近帖子。
4. 每读取一个帖子，就将其转换成内部标准导入 DTO。
5. 调用统一 Prisma 写库服务执行幂等导入。
6. 成功后推进帖子级游标，并更新 `importJobs` 统计。
7. 当前批次完成后继续拉取更旧帖子。
8. 失败时记录错误摘要并停在最近一次安全游标之后。
9. 下次继续任务时，从上一个已完成帖子之后继续。

## 游标与恢复语义

第一版不追求回复级恢复。

恢复语义应当是：

- 某个帖子全部成功写入后，才认为游标向前推进。
- 如果某个帖子处理失败，这个帖子不记为完成。
- 再次恢复时，从最后一个完成帖子之后重新开始。

这样设计可以保证：

- 不会把半个帖子的状态误记为已完成。
- 游标恢复逻辑更容易推导。
- 进度显示能稳定反映迁移范围。

## Admin 控制面

Admin 页面第一版至少应提供：

- 创建迁移任务按钮
- 最近迁移任务表格
- 当前状态展示
- 已处理帖子数
- 已处理回复数
- 最近更新时间
- 当前游标摘要
- 错误摘要
- 继续任务按钮
- 停止任务按钮

第一版不做复杂筛选和全文搜索，先确保最近任务的状态可见、可继续、可停止。

## 错误处理

迁移系统至少需要区分以下错误：

- 旧库连接失败
- 旧库查询失败
- 旧库数据字段不符合预期
- DTO 转换失败
- Prisma 写库失败
- 数据幂等冲突
- 任务运行中断

处理原则：

- 失败必须落库到 `importJobs.errorMessage`
- 必要时也应写入对应 `imports` 失败记录
- 前端 Admin 必须能直接看到失败摘要
- 不允许只在控制台打印错误后静默丢失任务状态

## 与后续部署的关系

当前采用同进程后台任务模式，因此部署时需要注意：

- Web 服务端必须是常驻进程。
- 后台任务状态必须写入数据库，而不是只存在内存。
- 容器重启后，后续实现需要决定是否自动恢复运行中的任务，或改为将其标记为可继续状态。

这部分属于实现策略，但设计边界已经要求不能依赖单次进程生命周期保存任务真相。

## 验证重点

这一子项目完成后，至少应验证：

1. Admin 可以创建旧库 `iwhisper` 迁移任务。
2. 任务能按“先新后旧”顺序导入帖子。
3. 帖子作为最小恢复单位时，失败后可以从上次完成位置继续。
4. 旧库作者按 `users.username` 正确归并为 `bot`。
5. 旧库数据可以成功转换成内部标准导入 DTO。
6. 旧库迁移与同步 API 导入可以复用同一套 Prisma 写库逻辑。
7. 迁移过程中，新库内容可以立即被公开阅读页消费。

## 非目标

这份设计明确不定义：

- 多进程或分布式迁移调度
- 回复级精细恢复
- 全板块迁移策略
- 最终迁移吞吐极限优化
- 任务实时推送或 WebSocket 进度流

这些内容应在迁移链路稳定后再单独细化。
