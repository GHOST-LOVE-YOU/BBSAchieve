# Idea Refine

## 项目背景

这个项目服务于一个学校论坛的匿名板块使用场景。

论坛中存在多个板块，其中“悄悄话”是活跃度很高的匿名板块。用户在这个板块中发帖或回复时，会以匿名编号而不是实名身份出现。为了支持后续的提醒、追踪、聚合等能力，后端需要先具备稳定、可复用、可扩展的数据采集能力。

当前 `backend` 的职责只聚焦在“采集”这一层，不负责前端页面、不负责通知、不负责业务数据库设计。

## 项目目标

当前阶段的目标：

1. 建立稳定的登录态管理能力。
2. 建立对论坛页面的最小可用采集能力。
3. 把不同采集能力拆成清晰模块，避免所有逻辑堆在一个入口里。
4. 为后续扩展帖子详情、增量抓取、时间过滤、更多板块支持预留结构。

非当前阶段目标：

1. 不在这里设计前端展示。
2. 不在这里设计通知系统。
3. 不在这里设计最终数据库结构。
4. 不在这里追求“一次性做完全部采集需求”。

## 需要知道的外部知识

这里只记录“对整个项目都成立”的领域知识，不记录某个具体功能的实现细节。

1. 目标站点是北邮人论坛，后端采集对象不是通用网站，而是一个有明确业务语义的校园论坛。
2. 论坛存在多个板块，“悄悄话”只是其中一个高优先级板块，不应把整个项目永久绑定为只能服务悄悄话。
3. 悄悄话中的匿名编号不是用户永久身份。
4. 悄悄话中的匿名编号范围是 `0-999`。
5. 多个真实用户可能在同一天被分配到相同匿名编号。
6. 悄悄话匿名编号会在每天早上八点刷新。

说明：

1. 某个具体接口怎么请求。
2. 某个页面的 DOM 长什么样。
3. 某个功能模块已经探索出的站点行为。

这类信息不要继续写在本文件里，而是下沉到对应子目录的 `AGENTS.md`。

## 文件结构

建议按“能力域”而不是“技术类型”组织。

当前结构：

```text
backend/
  AGENTS.md
  pyproject.toml
  uv.lock
  src/
    byr_auth/
    byr_boards/
```

当前职责划分：

1. `src/byr_auth/`
   负责登录态、cookie、会话检查等认证相关能力。
2. `src/byr_boards/`
   负责板块页抓取与解析。

后续推荐结构：

```text
backend/
  AGENTS.md
  pyproject.toml
  uv.lock
  src/
    byr_auth/
      AGENTS.md
    byr_boards/
      AGENTS.md
    byr_threads/
      AGENTS.md
    shared/
```

结构原则：

1. `byr_auth` 只处理认证，不掺杂板块和帖子解析。
2. `byr_boards` 只处理板块列表页，不掺杂帖子详情。
3. `byr_threads` 未来处理帖子详情、分页、楼层解析。
4. `shared` 只放多个能力域都会复用的通用内容。
5. 每个目录下如果已经积累了“只和该目录功能相关”的背景知识，就可以新增本地 `AGENTS.md`。
